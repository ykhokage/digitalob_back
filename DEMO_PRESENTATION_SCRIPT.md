# Сценарий рассказа для демонстрации проекта

Тема проекта: **«Разработка веб-приложения для мониторинга состояния микросервисной архитектуры с системой оповещений»**.

Название системы: **«Цифровой Наблюдатель»**.

## 1. Короткое вступление

В рамках дипломного проекта разработано веб-приложение для мониторинга микросервисной архитектуры. Система позволяет добавлять микросервисы, автоматически проверять их состояние, собирать технические метрики, фиксировать инциденты, отправлять уведомления и формировать отчеты по SLA/SLO.

Главная идея проекта заключается в том, чтобы администратор или пользователь системы мог в одном интерфейсе видеть состояние всех своих микросервисов, быстро понимать причину деградации и получать уведомления при проблемах.

## 2. Основной функционал

### Авторизация и пользователи

В системе реализованы регистрация и вход пользователя, подтверждение почты, восстановление пароля, refresh tokens и разграничение ролей.

Есть две роли:

- **ADMIN** — администратор;
- **OBSERVER** — обычный пользователь.

Обычный пользователь видит только свои микросервисы, инциденты, уведомления и отчеты. Администратор дополнительно имеет доступ к управлению пользователями, диагностике, журналу событий и демо-стенду.

### Микросервисы

Пользователь может добавлять микросервисы, указывая название, URL, тип, окружение и пороги мониторинга. Для каждого сервиса доступен «паспорт сервиса»: URL, статус, SLA, последние проверки, метрики, зависимости и связанные инциденты.

Система поддерживает разные состояния сервиса:

- норма;
- предупреждение;
- критическое состояние;
- отключен;
- нет данных.

### Мониторинг и метрики

Backend и отдельный worker регулярно проверяют сервисы по endpoint'ам:

- `/health`;
- `/liveness`;
- `/readiness`;
- `/metrics`.

Собираются метрики:

- доступность;
- время ответа;
- HTTP-статус;
- CPU;
- RAM;
- диск;
- количество запросов в минуту;
- ошибки приложения;
- uptime/downtime.

На frontend эти данные отображаются в виде карточек, графиков, таблиц и аналитических блоков.

### Инциденты

Если сервис становится недоступен или превышает заданные пороги, система автоматически создает инцидент. У инцидента есть критичность, статус, время начала, время завершения, описание и возможная причина.

Также предусмотрены комментарии к инцидентам и ручное закрытие после восстановления сервиса.

### Уведомления

Реализована система правил уведомлений. Пользователь может настроить, при каких событиях и куда отправлять уведомления.

Поддерживаются каналы:

- Email;
- Telegram;
- Web Push;
- Webhook;
- Max beta.

Для уведомлений реализованы дедупликация, тихие часы, журнал отправки и подтверждение получения.

### Карта архитектуры

Отдельный раздел показывает карту микросервисной архитектуры. На ней отображаются сервисы, зависимости между ними и влияние отказа одного сервиса на другие. Это помогает визуально объяснить, как сбой одного компонента может повлиять на всю систему.

### Отчеты

Система формирует отчеты по SLA/SLO за выбранный период. Отчеты можно экспортировать в PDF, Excel и CSV.

В отчетах отображаются:

- период отчета;
- список сервисов;
- доступность;
- среднее время ответа;
- количество инцидентов;
- SLA/SLO;
- графики и сводная аналитика.

Файлы отчетов и аватары пользователей сохраняются в объектное S3-хранилище.

### Демо-стенд

Для демонстрации работы проекта предусмотрен локальный стенд микросервисов. Сервисы можно переводить в разные режимы:

- нормальная работа;
- деградация;
- ошибки;
- полное падение;
- высокая CPU-нагрузка;
- высокая RAM-нагрузка;
- проблема с диском;
- резкий рост запросов.

Это позволяет на защите показать полный сценарий: сервис деградирует, появляется инцидент, меняется SLA, отправляется уведомление и меняется карта архитектуры.

## 3. Архитектура системы

Проект разделен на три независимых части:

1. **Frontend** — пользовательский интерфейс.
2. **Backend** — REST API, бизнес-логика, база данных, уведомления, отчеты.
3. **Microservices** — локальный стенд HTTP-микросервисов для демонстрации.

Общая схема работы:

```text
Пользователь
    |
    v
React Frontend
    |
    | REST API + JWT
    v
NestJS Backend API
    |
    | Prisma ORM
    v
PostgreSQL

Отдельный Worker
    |
    v
Проверка микросервисов -> Метрики -> Инциденты -> Уведомления

Дополнительные интеграции:
Redis / Upstash
Yandex Object Storage
Resend Email
Telegram Bot
Web Push
Webhook
Max beta
```

### Почему worker вынесен отдельно

Фоновый мониторинг вынесен в отдельный процесс, чтобы API не занималось тяжелыми периодическими проверками. Это ближе к production-подходу: API отвечает на запросы пользователя, а worker независимо собирает метрики, создает инциденты и отправляет уведомления.

### Почему данные пользователей изолированы

После регистрации каждый пользователь работает только со своими сервисами. Это важно, потому что система мониторинга может использоваться несколькими командами или организациями, и данные разных пользователей не должны смешиваться.

## 4. Технологический стек и обоснование выбора

### Frontend

**React** выбран для построения интерактивного SPA-интерфейса с большим количеством динамических состояний: графики, таблицы, фильтры, формы, модальные окна и dashboard.

**Vite** используется как быстрый инструмент разработки и сборки. Он быстрее поднимает dev-сервер и ускоряет разработку по сравнению с более тяжелыми сборщиками.

**TypeScript** повышает надежность кода: типизируются ответы API, доменные модели, свойства компонентов и формы.

**Tailwind CSS** выбран для быстрой и единообразной стилизации интерфейса. Он помогает поддерживать светлую и темную темы, адаптивность и единый дизайн.

**Recharts** используется для графиков метрик, SLA и динамики производительности. Библиотека хорошо подходит для React и позволяет быстро строить понятные интерактивные графики.

**Lucide React** используется для иконок, чтобы интерфейс был визуально аккуратным и единообразным.

**Playwright** используется для e2e-проверок интерфейса.

### Backend

**NestJS** выбран потому, что он хорошо подходит для модульных backend-приложений. В проекте есть отдельные модули: auth, users, services, metrics, incidents, notifications, reports, diagnostics, audit, worker. NestJS дает dependency injection, guards, decorators, validation pipes и понятную структуру.

**TypeScript** используется и на backend, и на frontend, поэтому проект получается единообразным и типобезопасным.

**Prisma ORM** выбран для работы с базой данных. Он дает типизированный клиент, миграции, удобную схему данных и снижает риск ошибок в SQL-запросах.

**PostgreSQL** выбран как надежная реляционная СУБД. Для проекта важны связи между пользователями, микросервисами, метриками, инцидентами, уведомлениями и отчетами, поэтому реляционная модель хорошо подходит.

**JWT + Refresh Tokens** используются для авторизации. Access token живет недолго, refresh token позволяет безопасно обновлять сессию.

**Guards и RBAC** используются для разделения прав администратора и обычного пользователя.

**class-validator** используется для проверки входных DTO, чтобы API принимал только корректные данные.

### Инфраструктура и интеграции

**Redis / Upstash** используется для кэша dashboard, дедупликации уведомлений и rate limit. Это снижает нагрузку на базу и предотвращает повторные уведомления по одному и тому же событию.

**Yandex Object Storage через S3 API** используется для хранения файлов: аватаров и экспортируемых отчетов. Это лучше, чем хранить файлы только локально, потому что хранилище не зависит от файловой системы сервера.

**Resend** используется для email-уведомлений и писем подтверждения.

**Telegram Bot API** используется как быстрый канал оперативных уведомлений.

**Web Push** позволяет отправлять уведомления прямо в браузер.

**Webhook** нужен для интеграции с внешними системами.

**Max beta** добавлен как расширяемый канал уведомлений.

### Локальные микросервисы

Стенд микросервисов написан на Node.js без тяжелых фреймворков, потому что его задача — быть легким, быстро запускаться и демонстрировать реальные HTTP endpoint'ы. Каждый сервис имеет собственный порт и возвращает реальные ответы для мониторинга.

## 5. Что показать на демонстрации

Рекомендуемый порядок:

1. Открыть страницу входа и войти под администратором.
2. Показать dashboard: общую доступность, метрики, последние события.
3. Перейти в «Микросервисы» и показать список сервисов.
4. Открыть паспорт одного микросервиса.
5. Показать карту архитектуры и зависимости.
6. Показать инциденты и объяснить, как они создаются автоматически.
7. Показать уведомления и каналы отправки.
8. Показать отчеты и экспорт PDF/XLSX/CSV.
9. Показать админ-разделы: пользователи, диагностика, журнал событий.
10. Запустить демо-сценарий или вручную перевести сервис в проблемное состояние.
11. Показать, что после деградации меняются метрики, появляется инцидент и отправляется уведомление.

## 6. Короткий вариант ответа на вопрос «что делает система»

Система «Цифровой Наблюдатель» предназначена для мониторинга микросервисной архитектуры. Она позволяет пользователям добавлять свои микросервисы, автоматически проверять их состояние, собирать технические метрики, отслеживать SLA, фиксировать инциденты, отправлять уведомления по разным каналам и формировать отчеты. Архитектура разделена на frontend, backend, отдельный worker и локальный стенд микросервисов, что делает проект близким к реальной production-системе.

## 7. Короткий вариант ответа на вопрос «почему такой стек»

React и Vite выбраны для быстрого и интерактивного frontend-интерфейса. NestJS выбран для модульного backend с понятной архитектурой, guards и dependency injection. PostgreSQL и Prisma используются для надежной реляционной модели и типобезопасной работы с данными. Redis нужен для кэша, дедупликации и rate limit. S3-хранилище используется для файлов отчетов и аватаров. Отдельный worker вынесен для фонового мониторинга, чтобы API оставался быстрым и не выполнял тяжелые периодические задачи внутри пользовательских запросов.

## 8. Возможные вопросы комиссии

**Почему не хранить метрики только на frontend?**

Потому что frontend не должен быть источником истины. Метрики собираются backend worker'ом, сохраняются в базу данных и доступны для отчетов, инцидентов и аналитики.

**Почему нужен отдельный worker?**

Проверка микросервисов — это периодическая фоновая задача. Если выполнять ее внутри API, это может ухудшить скорость ответов пользователям. Отдельный worker разделяет ответственность.

**Почему PostgreSQL?**

В системе много связанных сущностей: пользователи, сервисы, метрики, инциденты, уведомления, отчеты. Реляционная база хорошо подходит для таких связей и транзакционной целостности.

**Почему Redis?**

Redis используется для быстрых временных данных: кэша dashboard, ограничения частоты запросов и дедупликации уведомлений.

**Почему нужны роли?**

Роли отделяют обычных пользователей от администраторов. Это защищает административные функции: управление пользователями, диагностику и журнал аудита.

**Почему отчеты сохраняются в S3?**

Файлы отчетов и аватары не должны зависеть от локального диска сервера. Объектное хранилище проще масштабировать и использовать при развертывании.

**Есть ли заглушки?**

Для демонстрации используется отдельный стенд локальных микросервисов. Это не статические заглушки, а реальные HTTP-сервисы с endpoint'ами `/health`, `/liveness`, `/readiness` и `/metrics`, которые backend проверяет как обычные микросервисы.
